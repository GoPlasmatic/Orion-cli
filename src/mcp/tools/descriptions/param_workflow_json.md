JSON string of the workflow definition. Required fields: "name" (string), "tasks" (array of task objects). Optional: "workflow_id" (custom ID string), "priority" (higher runs first, default: 0), "condition" (JSONLogic, default: true), "description", "tags", "continue_on_error". Each task: {"id": "...", "name": "...", "function": {"name": "parse_json|parse_xml|map|filter|validation|http_call|log|publish_json|publish_xml|publish_kafka|channel_call", "input": {...}}}. Start most workflows with a parse_json task: {"name": "parse_json", "input": {"source": "payload", "target": "order"}}. Validation uses rules array: {"name": "validation", "input": {"rules": [{"logic": <JSONLogic>, "message": "..."}]}}. publish_json uses source/target: {"name": "publish_json", "input": {"source": "order", "target": "json_string"}}. http_call requires connector: {"name": "http_call", "input": {"connector": "name", "method": "POST", "path": "/api/...", "body_logic": ..., "response_path": "data.result"}}. channel_call invokes another channel in-process: {"name": "channel_call", "input": {"channel": "auth-service", "data": {...}, "response_path": "auth_result"}}. See the workflows_create tool description for full format details and examples.
